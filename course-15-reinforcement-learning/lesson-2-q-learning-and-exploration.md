*Course 15: Reinforcement Learning*

## Core Question

Q-learning has a beautiful convergence guarantee in the tabular case. So why does everything fall apart when you replace the table with a neural network, and what engineering tricks paper over the cracks?

---

## Q53: Why Q-Learning Is Unstable with Function Approximation

### Tabular Q-Learning: The Happy Case

In tabular Q-learning, you maintain a table Q(s,a) with one entry per state-action pair. After each transition (s, a, r, s'), you update:

```
Q(s,a) ← Q(s,a) + α [ r + γ max_{a'} Q(s', a') - Q(s,a) ]
```

The term in brackets is the **TD error** — the difference between the target `r + γ max Q(s',a')` and your current estimate. Under the Robbins-Monro conditions (learning rate decreases properly, every state-action pair visited infinitely often), this converges to Q* with probability 1. Clean, beautiful, proven.

Now replace the table with a neural network Q_θ(s,a). Train by minimizing:

```
L(θ) = E[ (r + γ max_{a'} Q_θ(s', a') - Q_θ(s,a))² ]
```

This is just regression, right? Fit the network to its own bootstrapped targets. What could go wrong? Three things, and they're all bad.

### Instability Source 1: Correlated Samples

In supervised learning, your training data is i.i.d. — each sample is drawn independently. In RL, your agent generates data by interacting with the environment sequentially. Consecutive transitions are heavily correlated: (s_1, a_1, r_1, s_2) is followed by (s_2, a_2, r_2, s_3). The network sees long stretches of similar states and overfits to the current region of the state space, forgetting what it learned about other regions.

**Fix — Experience Replay:** Store transitions in a large buffer. Sample mini-batches uniformly at random from the buffer. This breaks the temporal correlation. The network sees a mix of old and new experiences, approximating the i.i.d. condition that SGD needs.

The replay buffer also gives you sample efficiency: you use each experience multiple times instead of once. DQN's replay buffer held 1 million transitions.

### Instability Source 2: Non-Stationary Targets

Look at the loss function again. The target `r + γ max Q_θ(s', a')` depends on the same parameters θ you're updating. Every gradient step changes the target. It's like trying to hit a moving bullseye — you chase the target, the target moves, you chase again. This can oscillate or diverge.

In supervised learning, the labels are fixed. In Q-learning with a neural network, the labels are computed from the same network you're training. This creates destructive feedback loops.

**Fix — Target Network:** Keep a separate copy of the network, Q_{θ⁻}, frozen for N steps (typically N = 10,000). Use it to compute targets:

```
target = r + γ max_{a'} Q_{θ⁻}(s', a')
```

Update θ by gradient descent, but only copy θ → θ⁻ every N steps. The target is now stationary within each window. Polyak averaging (θ⁻ ← τθ + (1-τ)θ⁻ with small τ) is an even smoother variant.

### Instability Source 3: Maximization Bias

The `max` operator systematically overestimates Q-values. Here's why: suppose you have noisy estimates of Q(s', a') for all actions a'. Even if the true values are all zero, max of a bunch of noisy zeros is positive. You're selecting the action with the highest noise, not the highest value. This bias propagates through bootstrapping — overestimates feed into future targets, inflating values further.

Formally: E[max(Q₁, Q₂, ..., Q_n)] ≥ max(E[Q₁], E[Q₂], ..., E[Q_n]) by Jensen's inequality (max is convex).

**Fix — Double DQN:** Decouple action selection from action evaluation. Use the online network to pick the best action, but evaluate it with the target network:

```
a* = argmax_{a'} Q_θ(s', a')
target = r + γ Q_{θ⁻}(s', a*)
```

The online network selects; the target network evaluates. Since their errors are somewhat independent, the max bias is reduced.

### The Deadly Triad

Sutton identified the fundamental issue. Three ingredients that are individually fine but toxic in combination:

| Ingredient | What it does | Why you want it |
|---|---|---|
| Function approximation | Generalize across states | Can't have a table for every pixel configuration |
| Bootstrapping | Use your own estimates as targets | Monte Carlo is too high variance |
| Off-policy learning | Learn from data generated by a different policy | Sample efficiency |

Any two are fine. All three together can diverge. Q-learning with neural networks has all three. That's why DQN needed the three engineering fixes above to work at all.

> **Key takeaway:** Deep Q-learning works not because we solved the theoretical problems, but because we found engineering patches (replay, target networks, double Q) that make the divergence manageable in practice. The theory still says it can diverge. The practice says "usually it doesn't, if you're careful."

## Q54: Exploration vs Exploitation — The Math

### The Problem

Your agent has a current best action. Should it do that (exploit) or try something else to see if there's something better (explore)? Get the balance wrong and you either waste time on known-bad options or miss something great you never tried.

### The Multi-Armed Bandit: Simplest Version

Strip away states, transitions, and time. You have K slot machines (arms). Each pull gives a random reward from an unknown distribution. You want to maximize total reward over T pulls. This isolates the explore/exploit tension in its purest form.

### Strategy 1: ε-Greedy

```
With probability 1-ε: pick argmax_a Q(a)     (exploit)
With probability ε:   pick a random action    (explore)
```

Simple. Universal. And wasteful. When you explore, you pick uniformly at random — the action you've pulled 1,000 times and the one you've pulled once are equally likely. You explore the worst action as eagerly as the second-best. Typically ε is annealed from a high value (1.0) down to a low value (0.01) over training.

### Strategy 2: Upper Confidence Bound (UCB)

```
a_t = argmax_a [ Q(a) + c · √(ln(t) / N(a)) ]
```

Where Q(a) is the estimated value, t is the total number of pulls so far, and N(a) is how many times you've pulled arm a. The second term is a **confidence bonus** — large when N(a) is small (you haven't tried this arm much) and shrinks as you gather data.

The principle is **optimism in the face of uncertainty**: assume unknown actions are good until proven otherwise. Unlike ε-greedy, UCB directs exploration toward undervisited actions, not random ones. It achieves logarithmic regret — the theoretical lower bound (up to constants) for the bandit problem.

### Strategy 3: Thompson Sampling

```
For each arm a:
    Sample θ_a ~ Posterior(Q(a) | observed rewards)
Pick a = argmax_a θ_a
```

Maintain a Bayesian posterior over each arm's reward distribution. Sample from each posterior, then act greedily on the samples. Arms you're uncertain about have wide posteriors — sometimes their sample will be high, causing exploration. Arms you know well have tight posteriors — they only win if their mean is actually high.

Thompson Sampling is often empirically optimal and has strong theoretical guarantees (Bayesian regret bounds). No ε or c hyperparameter to tune.

### Comparison

| Method | Explores | Regret | Practical | Handles Uncertainty |
|---|---|---|---|---|
| ε-greedy | Randomly | O(ε·T) for fixed ε | Very easy | No |
| UCB | Undervisited arms | O(√(KT ln T)) | Medium | Yes (frequentist) |
| Thompson Sampling | Uncertain arms | O(√(KT ln K)) | Medium | Yes (Bayesian) |

### Why RL Exploration Is Harder Than Bandits

In bandits, each action gives immediate, independent feedback. In full RL:

1. **Delayed rewards:** The exploratory action might not pay off for thousands of steps. You don't get immediate feedback about whether exploration was worth it.
2. **State-dependent:** A good exploration strategy in state s₁ is terrible in state s₂. You need to explore the *joint* state-action space.
3. **Compounding:** A single bad exploratory action can land you in a state from which recovery is impossible (or very expensive).

Modern deep RL uses **intrinsic motivation** (bonus reward for visiting novel states), **curiosity-driven exploration** (reward proportional to prediction error of a learned dynamics model), and **count-based methods** (approximate visitation counts in high-dimensional state spaces using density models). These are all heuristics — the exploration problem in general RL is fundamentally harder than in bandits, and no algorithm achieves the theoretical lower bound except in special cases.

> **Key takeaway:** Exploration is about reducing uncertainty efficiently. ε-greedy is the sledgehammer; UCB and Thompson Sampling are the scalpels. In deep RL, the exploration problem is so hard that most practical systems rely on simple ε-greedy plus some form of intrinsic reward, because the theoretically elegant methods don't scale to high-dimensional state spaces.

## Q&A

**Question:** An interviewer says: "You mentioned the deadly triad. Give me a concrete scenario where only two of the three ingredients are present, and explain why it's stable."

**Student's Answer:** "Monte Carlo control with a neural network. You use function approximation (ingredient 1) and off-policy learning via importance sampling (ingredient 3), but you don't bootstrap — you use full episode returns as targets. Without bootstrapping, the target is a fixed quantity (the actual observed return), not a moving estimate. So you're doing standard supervised regression with a fixed target, just with importance-weighted samples. The gradient points in a consistent direction, and SGD can converge. The dangerous feedback loop — where your estimate feeds into your target which feeds into your estimate — is broken."

**Evaluation:** Excellent. The student correctly identified that removing bootstrapping eliminates the self-referential target problem. Monte Carlo methods with function approximation can still have issues (high variance from importance weights, for instance), but they don't have the divergence pathology of the deadly triad. This is exactly the kind of precise, mechanistic reasoning interviewers want.

**Question:** In Double DQN, we use the online network to select the action and the target network to evaluate. Why not the reverse?

**Student's Answer:** "The point is to decouple selection and evaluation so their noise sources are different. If you used the target network to select and the online network to evaluate, it would still decouple them — but the target network is stale (it's a frozen old copy), so it would select actions based on outdated information. The online network has the freshest estimates, so it makes better action selections. You want the best selector paired with the most independent evaluator. Since the target network is already a separate (if old) copy, it provides the independence you need for evaluation without degrading selection quality."

**Evaluation:** Spot on. The asymmetry matters practically even though the theoretical debiasing works either way. The online network is more current and makes better selections; the target network provides independent evaluation. Reversing the roles would work in theory but give worse empirical performance because the stale target network would frequently select suboptimal actions.
