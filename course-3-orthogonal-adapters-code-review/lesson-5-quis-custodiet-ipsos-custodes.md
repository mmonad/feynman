# Lesson 5: Quis Custodiet Ipsos Custodes?

*Course 3: Orthogonal Adapters & Composable Code Review Committees*

*"Who watches the watchmen?" — Juvenal, roughly 2,000 years before anyone worried about AI code review.*

## The Self-Referential Trap

The committee is a system of LLMs reviewing code written by LLMs. The composition model is an LLM evaluating LLMs' opinions. Trained using synthetic data generated by LLMs injecting mutations that LLMs designed.

At what point does this become a **closed loop** — a system that can only see what systems like it can see?

We solved the *first-order* blind spot problem (single model reviewing its own code). Our orthogonal adapters solved the *second-order* problem (multiple models sharing blind spots). But there may be **third-order blind spots** — things *no LLM architecture* can see, regardless of training or adaptation.

## What Could All LLMs Miss?

### 1. Specification-Level Errors

Code does exactly what it's told. Clean, secure, performant. But **what it's told to do is wrong.** Tax brackets changed last quarter, nobody updated the spec. LLMs review code against patterns and logic — not against **the current state of business reality**.

### 2. Architectural Misfit

Code is beautiful in isolation but solves the problem at the wrong layer. Caching in the application when it should be at the CDN. Retry logic in the client when the server should be idempotent. LLMs evaluate code *locally* but are weak at evaluating whether code **should exist at all** in its current location.

### 3. Emergent Runtime Behavior

Code is correct under static analysis. But under production load — 10,000 concurrent users, GC pressure, network partitions — emergent behaviors arise **invisible to any model**. Race conditions, memory leaks over hours, cascading failures. Properties of the running system, not the source code.

### 4. Social and Organizational Bugs

"The upstream team is about to deprecate that API." "This violates an unwritten convention the platform team will reject." "Three other teams are making conflicting similar changes." Bugs in **organizational context** no model can access.

## The Committee's True Role

```
                  Bugs in code
                 /            \
    +-----------------+   +-----------------+
    |  Things LLMs    |   |  Things LLMs    |
    |  CAN catch      |   |  CANNOT catch   |
    |                 |   |                 |
    | - Logic errors  |   | - Spec errors   |
    | - Security      |   | - Architectural |
    |   vulnerabilities|   |   misfit        |
    | - Performance   |   | - Emergent      |
    |   anti-patterns |   |   runtime bugs  |
    | - API misuse    |   | - Organizational|
    | - Style/maintain|   |   context bugs  |
    +-----------------+   +-----------------+
           |                      |
    Committee handles          Still needs
    this VERY well             humans
```

The committee doesn't replace human review. It handles the left column so humans focus exclusively on the right column — high-judgment, context-dependent decisions. Currently, human reviewers spend ~80% of time on left-column work. Freeing them is a massive efficiency gain.

## The Monitoring Layer: Watching the Watchmen

### Signal 1: Escape Rate Tracking

Monitor bugs reaching production that the committee should have caught:

```
Month 1:  2 escaped    Month 3:  3 escaped
Month 2:  1 escaped    Month 4:  7 escaped  <- adapters need retraining
```

### Signal 2: Human Override Rate

```
Human overrides "approve" → committee missed something     (false negative)
Human overrides "flag"    → committee was too aggressive    (false positive)
```

Rising false negatives = dangerous (losing sensitivity). Rising false positives = annoying (wasting human attention). Both indicate *which adapters* need retraining.

### Signal 3: Inter-Adapter Agreement Drift

Orthogonal adapters should disagree at a healthy, stable rate. Sudden increase in agreement → adapters may be converging (losing orthogonality). Sudden spike in disagreement → miscalibration. Track disagreement rate over time — sudden changes are diagnostic.

### Signal 4: Mutation Test Regression

Periodically run mutation testing pipeline from Lesson 3. If catch rate drops on fresh mutations, committee is degrading. The most direct measure — a unit test for the review system itself.

## The Full System: Defense in Depth

```
Layer 1:  Orthogonal specialist adapters
          (catch single-dimension bugs)
              |
Layer 2:  Composition model
          (catch diagonal/cross-cutting bugs)
              |
Layer 3:  Confidence calibration + disagreement analysis
          (catch committee miscalibration)
              |
Layer 4:  Human reviewers
          (catch spec errors, architectural misfit,
           organizational context bugs)
              |
Layer 5:  Production monitoring + incident traces
          (catch everything else — feedback loop
           to retrain Layers 1-3)
```

Each layer handles a different class of problem. No single layer sufficient. Together, covers far more than any one approach. And it's a **loop** — Layer 5 feeds back into Layers 1-3. The continual learning loop from Course 2 applied to the entire review system.

## The Final Insight

> **Intelligence — whether human or artificial — is not about having the right answer. It's about having the right *diversity of perspectives* and the right *process for integrating them.***

A single brilliant reviewer < a committee of orthogonal specialists with good composition. A single powerful model < a system of adapted models covering each other's blind spots. The whole AI system < AI + humans providing perspectives no model can.

"Who watches the watchmen?" → **Everyone watches each other, from orthogonal angles, and the system is designed so no single failure goes undetected.**

Not just a code review architecture. A design principle for building reliable systems from unreliable components.

### The Kegan Parallel

Kegan's stages of adult development describe how humans grow in ability to hold complexity:

```
Stage 1: Impulsive         → single fixed response to stimuli
Stage 2: Imperial          → own perspective, can't hold others'
Stage 3: Socialized        → can hold multiple perspectives, defined by them
Stage 4: Self-Authoring    → integrates perspectives into own coherent framework
Stage 5: Self-Transforming → framework itself evolves through dialectic
```

The AI system we built:

```
Static training          → single fixed model (Stage 2: one perspective)
LoRA adapters            → specialization, still isolated (Stage 2-3)
Orthogonal adapters      → multiple guaranteed-diverse perspectives (Stage 3)
Composition model        → integrates perspectives into coherent judgment (Stage 4)
Continual feedback loop  → system evolves its own frameworks over time (Stage 5)
```

The mapping is **structural, not metaphorical.** Each stage solves the same problem: moving from a single fixed viewpoint to a self-evolving system that integrates diverse perspectives and grows from its own contradictions.

### The Societal Parallel

Individual craftsmen → guilds with diverse specialties → democratic deliberation → institutional feedback loops → evolving constitutions.

Societies figured out that no single perspective governs well. They invented structured disagreement (parliaments), conflict resolution (courts), self-correction (constitutional amendments). Slowly, painfully, over centuries.

We're building the same structures for AI. Orthogonal adapters = parliamentary seats. Composition model = deliberation process. Confidence calibration = evidentiary standard. Feedback loop = constitutional amendment process. In months, not millennia.

Whether that's an intelligence explosion or the same old pattern of complexity finding its footing — the fact that it *looks like* history repeating is either deeply reassuring (we know how this goes) or deeply unsettling (we also know how it goes wrong).
